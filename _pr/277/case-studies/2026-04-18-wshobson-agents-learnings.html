wshobson/agents — debrief ​

Companion to 2026-04-18-wshobson-agents.md. That one is the public artifact, written by the automated pipeline. This one is the internal debrief: what the engagement taught us about NLPM itself, what we got wrong, and what should change before the next run.

Audience: xiaolai and anyone else building contribution pipelines.

The honest timeline ​

The auto-generated article reports "five PRs merged in 12 seconds." That's accurate but misleading in isolation. Full wall clock:

PhaseStartEndDuration
NLPM audit2026-04-17 12:24:1912:33:309m 11s (active compute)
Gap — batch processor poll12:33:3012:49:5916m 29s (idle)
NLPM contribute (5 PRs)12:49:5912:56:246m 25s (active compute)
Maintainer response latency12:55:28 (last PR open)17:15:22 (first merge)4h 19m 54s (idle)
Merge burst (5 PRs)17:15:2217:15:3513 seconds

So: ~15 min of active compute on our side, ~31 min wall clock from onboarding to PRs live, then 4h 20m of nothing, then 13 seconds of merging. "Instant" it was not. "Fastest we've seen" is true only inside a narrow window.

Estimated Claude-token cost for the audit + contribute: $0.25–$0.80 (Sonnet tier, ~30K–50K tokens across both calls). Cheap; the polling gap is more expensive than the compute.

The agentic handshake ​

Evidence both sides were running pipelines:

SignalWhat it suggests
Merge cadence 17:15:22, :25, :29, :32, :35 — 3s per PRScripted merge loop, not tab-clicking
5 identical comments: "Thanks @xiaolai — merged. Appreciate the NLPM audit catching these silent registration failures."Template, not typing

The likely shape: wshobson's side runs a scheduled agentic sweep (4h response latency suggests a cron-like trigger) that applies a rubric — small diff + CI-green + NLPM disclosure present + mechanical pattern match — and batches the merges. It may be fully autonomous or click-confirm-batch; from the outside they look the same.

Our disclosure at the top of each PR body worked. It's becoming protocol:

> **Automated audit**: This PR was generated by [NLPM](https://github.com/xiaolai/nlpm-for-claude), a natural language programming linter, running via [claude-code-action](https://github.com/anthropics/claude-code-action). Please evaluate the diff on its merits.

Agentic reviewers on the other side can pattern-match this and route accordingly. Humans can too. We should keep it.

The sampling blind spot ​

ScopeBroken commandsBroken agentsTotal

Full repo (282 files)50757
Fixed by merged PRs1313
Still broken after merges41344

The $MAX_FILES=100 cap we added three days ago to survive ARG_MAX on large repos also hid two-thirds of this specific bug. We shipped 5 PRs that fix a real problem — and left 44 files of the same problem untouched. If a human reviewer did this they'd be called negligent.

The cap isn't wrong; unbounded file lists break the workflow. But a flat cap is the wrong granularity. When the audit finds the same bug pattern on >5 files, it should trigger a second pass scoped to just that pattern, without the 100-file cap, because grepping for one pattern across 500 files is cheap.

Meta bugs in NLPM this engagement surfaced ​

Three bugs in our own workflows found and fixed while doing this audit. Listed because they will recur on the next big repo if we forget:

  1. git checkout --theirs silently dropped workflow's own data on push-retry conflicts. The contribute workflow wrote pipeline_prs: [488-492] to the registry; a concurrent push raced; the retry merged in the remote and used --theirs which replaced our new data with the stale remote. Track workflow then couldn't find the PRs, no case-study-ready label fired, no case study wrote. Fix (81fbc7d): auditor/scripts/resolve-merge-conflicts.sh does jq deep-merge for repos.json, line-union for events.jsonl, --ours for everything else. Applied to audit + contribute + daily-report workflows.

  2. Track workflow added case-study-ready label but case-study workflow never fired. GitHub suppresses issues: labeled events from GITHUB_TOKEN (anti-recursion). Same pattern batch-processor hit for audit-ready last week. Fix (81fbc7d): track now calls gh workflow run auditor-case-study.yml explicitly after labeling. Requires actions: write permission.

  3. Case-study workflow had no push retry. Single git push, no retry loop. Raced my own fix commit and failed; article + cover image abandoned on the destroyed runner. Had to re-dispatch. Fix (50d2f84): same 5-attempt retry loop as the others.

All three are race bugs. The pipeline's per-workflow isolation masks them in most runs; they surface under load. We should assume every workflow that pushes will race eventually.

Proposals (ranked by expected impact) ​

Each has a concrete entry point in the codebase.

1. Same-pattern full-scan trigger ​

Where: auditor-audit.yml after the main scoring pass. When any bug category has N>5 instances in the sample, grep the full repo for that category's signature and append findings. Bound by a grep-based cap (cheap), not a Claude-call cap.

Example rule: if BUG-missing-frontmatter count >= 5, run find . -path '*/agents/*.md' -o -path '*/commands/*.md' | while read f; do head -1 "$f" | grep -qx '---' || echo "$f"; done and append each result as a bug finding without re-reading the file.

Expected impact: would have caught all 57 files on wshobson/agents in the first pass.

2. Reviewer-type tagging in the registry ​

Where: auditor/registry/repos.json — new field reviewer_type: agentic | human | hybrid | unknown inferred from signals: PR merged within N seconds of prior PR on same day, template comments, presence of claude-*.yml workflows in target repo, Co-authored-by: Claude in target's commit history.

Why it matters: human-velocity and agent-velocity are different signals. Conflating them makes "merge speed" useless as a prioritization input. Once split, the pipeline can route agentic-reviewer repos to an optimized PR template and prioritize them for backlog drain.

3. Split the maintainer-velocity metric ​

Where: registry field maintainer_velocity → replace with maintainer_response_latency (PR-open → first-merge-action) and merge_burst_duration (first-merge → last-merge-of-batch). Both computed by track workflow from existing .prs data.

Why: wshobson is 4h 20m on response latency (merely attentive) and 13s on burst (unmistakably agentic). Both numbers are useful; the average is not.

4. Keep the disclosure-at-PR-top convention; document it ​

Where: skills/nlpm/patterns/ — add a pattern doc for agent-to-agent PR disclosure. The current boilerplate works; it should be explicit in the skill, not just in the contribute workflow's prompt.

5. Close the cross-workflow race loophole ​

Where: all 5 auditor workflows. Any bot push should assume race. The retry loop is now consistent across audit, contribute, daily-report, case-study. Track still has no retry loop — harmless today because track runs once daily and reads mostly, but should gain the same loop before we ever make track write new data under load.

Open questions ​

Things this engagement doesn't answer and shouldn't claim to:

  1. Was wshobson's merge fully autonomous, click-confirm, or something else? We can't tell from outside. The signals are consistent with scripted merge after agentic review. Could also be a human doing a rapid visual spot-check. Likely doesn't matter for our purposes — the rubric matters, not the agency.

  2. Why did the batch processor leave a 16-minute poll gap? Audit finished 12:33; contribute started 12:49. That gap is longer than either compute phase. Probably an artifact of the 6-hour batch schedule catching the audit at a suboptimal moment. Worth measuring across all past engagements before adjusting the schedule.

  3. Is agentic review the majority case on large Claude Code plugin repos yet, or still an outlier? wshobson is one data point. If we tag reviewer_type across the registry's 58 repos we'll have a base rate to compare.

What to do next, in order ​

  1. Land proposal #1 (same-pattern full-scan trigger) — highest leverage; would have changed the wshobson outcome from "fixed 13 of 57" to "fixed 57 of 57" with the same compute budget.
  2. Land proposal #2 (reviewer-type tagging). Pure observation pass, no workflow changes needed to start — just grep existing registry PR data.
  3. Draft the 8 per-plugin PRs for the remaining 44 wshobson files, hold until discussion #487 gets a response or until a week passes without one.
  4. Re-run proposal #1 against the other 55 audited repos and see how many have the same hidden-scope problem. Guess: most.
